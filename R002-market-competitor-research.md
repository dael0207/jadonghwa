# R002 — Market, Competitor & Use-Case Research

## 1. 문서 목적

이 문서는 실제 업무 자동화 시장과 경쟁 제품의 접근 방식을 분석하고, Work Discovery AI가 어떤 문제를 해결해야 하며 어디에서 차별화해야 하는지를 정의한다.

조사 범위는 실제 업무 자동화 사례, 프로세스 마이닝, 태스크 마이닝, 워크플로 자동화 플랫폼, RPA, AI Agent Builder, 자동화 컨설팅, 업무 발견 방식, 자동화 후보 선정 방식, 실제 프로그램 제작 연결성이다.

## 2. 시장의 주요 문제

업무 자동화 도구는 이미 많지만 다음과 같은 공백이 존재한다.

- 사용자가 무엇을 자동화해야 하는지 모른다.
- 사용자가 자신의 업무 흐름을 정확하게 설명하지 못한다.
- 화면 행동은 수집할 수 있어도 판단 이유와 예외는 파악하기 어렵다.
- 업무 설명을 바로 워크플로로 만들면 빠진 조건 때문에 불완전한 자동화가 생긴다.
- 자동화 이전에 제거하거나 단순화해야 할 업무를 구분하지 못한다.
- 전문 컨설팅은 비용과 시간이 많이 든다.
- 자동화 기회 발견과 실제 개발 명세 사이가 끊겨 있다.

## 3. 경쟁 시장 분류

### 3.1 프로세스 마이닝

대표 제품:

- Celonis
- Microsoft Power Automate Process Mining
- SAP Signavio
- UiPath Process Mining

#### 접근 방식

ERP, CRM, MES, 그룹웨어 등에서 발생한 이벤트 로그를 분석해 실제 프로세스 흐름을 재구성한다.

```text
주문 생성
→ 결재 요청
→ 결재 완료
→ 출고
→ 대금 수령
```

#### 강점

- 실제 시스템 데이터를 기반으로 분석한다.
- 병목, 반복, 지연, 재작업을 정량화할 수 있다.
- 조직 단위 프로세스 분석에 적합하다.
- 프로세스 변형과 표준 절차 이탈을 찾을 수 있다.

#### 한계

- 시스템 로그가 존재해야 한다.
- 이메일, 전화, 회의, 개인 파일 등 비공식 업무를 놓칠 수 있다.
- 행동의 이유와 판단 기준을 파악하기 어렵다.
- 소규모 조직이나 개인에게는 도입 부담이 크다.
- 업무 수행자의 암묵지를 직접 얻기 어렵다.

#### 시사점

- 장기적으로 시스템 로그와 연결할 수 있어야 한다.
- 초기 MVP는 로그 없이도 업무를 분석할 수 있어야 한다.
- 인터뷰 결과와 실제 로그가 충돌할 수 있으므로 근거 상태를 분리해야 한다.

### 3.2 태스크 마이닝

대표 제품:

- UiPath Task Mining
- Microsoft Task Mining
- Automation Anywhere Process Discovery
- ABBYY Task Mining
- Skan

#### 접근 방식

사용자의 PC 작업, 애플리케이션 전환, 클릭, 입력, 복사·붙여넣기 등을 기록해 반복 작업을 찾는다.

#### 강점

- 사용자가 기억하지 못하는 실제 작업을 발견할 수 있다.
- 앱 전환, 반복 클릭, 복사 작업 등을 정량화할 수 있다.
- 작업 시간과 빈도를 비교적 객관적으로 계산할 수 있다.
- 여러 직원의 작업 흐름을 비교할 수 있다.

#### 한계

- 개인정보와 직원 감시 우려가 크다.
- 화면 밖의 판단, 회의, 전화 업무를 파악하기 어렵다.
- 사용자가 왜 특정 행동을 했는지 알기 어렵다.
- 업무 예외와 숙련자의 판단 기준을 이해하기 어렵다.

#### 시사점

- 초기에는 상시 화면 녹화를 제외한다.
- 행동 정보보다 판단·예외·목적을 우선 수집한다.
- 장기적으로 사용자가 선택한 특정 업무 시연만 증거로 추가할 수 있다.
- 직원 평가나 감시 목적 사용을 명시적으로 제한해야 한다.

### 3.3 워크플로 자동화 플랫폼

대표 제품:

- Zapier
- Make
- n8n
- Microsoft Power Automate
- Gumloop

#### 접근 방식

사용자가 트리거, 조건, 액션을 직접 구성하거나 자연어로 자동화를 요청한다.

```text
이메일 도착
→ 첨부파일 저장
→ 데이터 추출
→ 스프레드시트 입력
→ 알림 전송
```

#### 강점

- 실제 자동화 실행까지 빠르게 연결할 수 있다.
- 다양한 SaaS와 API를 연결할 수 있다.
- 반복 업무에 대한 즉각적인 가치를 제공한다.

#### 한계

- 사용자가 자동화 대상을 이미 알아야 한다.
- 예외와 판단 기준을 사용자가 직접 정의해야 한다.
- 업무가 충분히 표준화되지 않으면 불완전한 플로가 만들어진다.
- 자동화보다 기존 기능 활용이나 프로세스 개선이 더 적합한 경우를 잘 구분하지 못한다.

#### 시사점

- 최종 산출물을 이러한 도구에 전달 가능한 형태로 만들어야 한다.
- Work Discovery AI는 실행 도구와 경쟁하기보다 앞단의 발견·설계 계층에 위치할 수 있다.

### 3.4 RPA 플랫폼

대표 제품:

- UiPath
- Automation Anywhere
- Blue Prism
- Power Automate Desktop

#### 접근 방식

사람이 화면에서 수행하는 반복 작업을 소프트웨어 로봇이 대신 실행한다.

#### 강점

- API가 없는 기존 시스템에도 적용할 수 있다.
- 정형화된 데스크톱 업무 자동화에 강하다.
- 기업용 권한, 모니터링, 운영 기능이 잘 갖춰져 있다.

#### 한계

- 화면 변경에 취약할 수 있다.
- 예외가 많거나 판단이 복잡하면 유지보수가 어려워진다.
- 자동화 후보 선정과 요구사항 정의가 별도 컨설팅에 의존하는 경우가 많다.

#### 시사점

- 자동화 수단 중 하나로 RPA를 추천할 수 있어야 한다.
- API 연동, 기존 기능 활용, RPA, 전용 앱을 비교해야 한다.

### 3.5 AI Agent Builder

대표 제품:

- Microsoft Copilot Studio
- Salesforce Agentforce
- Lindy
- Relevance AI
- OpenAI 기반 커스텀 에이전트 제품

#### 접근 방식

자연어로 에이전트의 목표, 도구, 지식, 실행 규칙을 정의한다.

#### 강점

- 비정형 문서와 자연어 업무에 대응할 수 있다.
- 여러 도구를 연결하고 순차 작업을 수행할 수 있다.
- 초안, 요약, 분류, 검색, 의사결정 보조에 강하다.

#### 한계

- 입력된 업무 설명이 부정확하면 에이전트도 잘못 설계된다.
- 권한, 책임, 예외, 실패 복구가 명확하지 않으면 위험하다.
- 자율성 수준을 과도하게 높일 가능성이 있다.

#### 시사점

- Agent를 추천하기 전에 사람의 승인 범위와 위험을 분석해야 한다.
- 정보 제공, 초안, 추천, 승인형 실행, 제한적 자동 실행을 구분해야 한다.

### 3.6 자동화 컨설팅

#### 접근 방식

컨설턴트가 인터뷰, 워크숍, 설문, 현장 관찰, 업무 목록화, ROI 분석을 통해 자동화 후보를 찾는다.

```text
직원 인터뷰
→ 업무 목록 작성
→ 반복성·시간·오류 분석
→ 자동화 후보 선정
→ ROI 분석
→ 개발팀 전달
```

#### 강점

- 조직 맥락, 판단, 예외, 책임을 이해할 수 있다.
- 자동화보다 업무 개선이 필요한 경우를 구분할 수 있다.
- 이해관계자 조정과 변화관리를 지원할 수 있다.

#### 한계

- 비용이 높다.
- 시간이 오래 걸린다.
- 컨설턴트 역량에 따라 품질 차이가 크다.
- 소규모 회사와 개인이 접근하기 어렵다.
- 인터뷰 결과가 개발 가능한 구조로 남지 않는 경우가 있다.

#### 시사점

- AI가 자동화 컨설턴트의 일부 역할을 구조화·반복 가능하게 만들어야 한다.
- 고위험·복잡한 업무는 전문가 검토로 넘겨야 한다.

## 4. 실제 업무 자동화 사례

### 4.1 송장·청구서 처리

```text
이메일로 송장 수신
→ 첨부파일 다운로드
→ 공급자·금액·날짜 확인
→ 주문서와 대조
→ ERP 입력
→ 예외 건 확인
→ 승인 요청
→ 문서 보관
```

자동화 가능 부분:

- 이메일 첨부파일 수집
- OCR 및 문서 분류
- 필드 추출
- 주문서와 금액 비교
- ERP 입력
- 누락·불일치 감지
- 예외 건 담당자 전달

사람 유지 부분:

- 신규 거래처 판단
- 비정상 금액의 최종 승인
- 계약·세금 관련 특수 판단

### 4.2 고객 문의 처리

```text
이메일·채팅 문의 수신
→ 문의 유형 확인
→ 고객 정보 조회
→ 답변 검색
→ 답변 작성
→ 담당 부서 전달
→ 처리 결과 기록
```

자동화 가능 부분:

- 문의 분류
- 고객 정보 검색
- 관련 문서 검색
- 답변 초안
- 티켓 생성
- 담당자 배정

사람 유지 부분:

- 감정적 고객 대응
- 환불·보상 승인
- 법적 분쟁 가능성이 있는 문의

### 4.3 월간 보고서 작성

```text
여러 시스템에서 데이터 다운로드
→ 엑셀 정리
→ 전월·전주 데이터 비교
→ 그래프 생성
→ 특이사항 작성
→ 보고서 파일 생성
→ 이메일 발송
```

자동화 가능 부분:

- 데이터 수집
- 형식 통합
- 계산
- 전월 비교
- 그래프 생성
- 이상치 표시
- 보고서 초안

사람 유지 부분:

- 이상 원인 확인
- 경영적 해석
- 최종 검토와 발송

### 4.4 규제·품질 문서 검토

```text
문서 수령
→ 문서 종류 확인
→ 필수 항목 확인
→ 이전 버전 비교
→ 누락·불일치 탐지
→ 수정 요청
→ 승인 및 보관
```

자동화 가능 부분:

- 문서 분류
- 체크리스트 검사
- 필수 항목 추출
- 버전 비교
- 누락 탐지
- 검토 의견 초안

사람 유지 부분:

- 최종 적합성 판단
- 규제 해석
- 책임이 수반되는 승인

## 6. 시장 공백

현재 시장에서 상대적으로 부족한 영역은 다음과 같다.

```text
자동화 지식이 없는 사용자
→ 자신의 업무 설명
→ AI의 반복 질문
→ 판단·예외·암묵지 구조화
→ 자동화 이전의 제거·단순화 판단
→ 자동화 후보 비교
→ 개발 가능한 프로그램 명세
```

이 공백이 Work Discovery AI의 핵심 시장 위치다.

## 7. 제품 차별화 방향

### 7.1 자동화 대상을 모르는 상태에서 시작

기존 제품은 “무엇을 자동화할 것인지”를 입력받는다. 우리 제품은 “무슨 일을 하는지”부터 묻는다.

### 7.2 질문을 통한 암묵지 수집

단순 설문이 아니라 실제 사례, 어려운 판단, 예외, 실패, 숙련자의 단서를 반복 질문한다.

### 7.3 근거 기반 업무 모델

사용자 발언, AI 추론, 문서 증거, 사용자 확인 상태를 구분한다.

### 7.4 자동화 이전의 업무 개선

삭제, 통합, 순서 변경, 단순화, 표준화, 기존 도구 활용을 먼저 검토한다.

### 7.5 자동화 수준 구분

- 정보 제공
- 초안 생성
- 추천
- 사람 승인 후 실행
- 제한적 자동 실행
- 완전 자동화

### 7.6 개발 명세까지 연결

추천 보고서에서 끝나지 않고 PRD, 화면, 데이터, API, 테스트, 개발 백로그를 생성한다.

## 8. 리서치에서 도출된 설계 요구사항

### 8.1 인터뷰만으로 충분하다고 가정하지 않는다

향후 다음 증거를 연결할 수 있어야 한다.

- 사용자 설명
- 음성
- 업무 문서
- PDF
- 스프레드시트
- 이메일
- 화면 캡처
- 선택적 업무 시연
- 시스템 로그

### 8.2 업무와 작업을 구분한다

```text
업무: 월간 매출 보고서 작성
```

## 9. 초기 시장 가설

1. 자동화에 관심은 있지만 자동화 대상을 모르는 중소기업 직원과 팀이 존재한다.
2. 사용자는 직접 프로세스 문서를 작성하는 것보다 질문에 답하는 방식을 더 쉽게 느낀다.
3. 자동화 후보만 제시하는 것보다 실제 프로그램 설계 명세까지 제공할 때 지불 의향이 높아진다.
4. 범용 제품으로 시작하더라도 제조·품질·규제, 회계, 영업 운영 등 세로형 질문 팩이 필요하다.
5. 완전 자동화보다 AI 보조와 승인형 자동화가 초기 도입에 더 적합한 경우가 많다.

## 10. 초기 검증 대상

첫 파일럿은 다음 업무 유형을 포함해야 한다.

1. 규칙 기반 반복 업무
2. 문서·메일 중심 업무
3. 전문 판단 중심 업무
4. 자동화 가치가 낮은 저빈도 업무
5. 개인정보·재무·규정 위험이 큰 업무
6. 회의·전화·물리 작업이 섞인 업무

## 11. MVP 전략에 대한 결론

첫 MVP는 범용 자동화 실행 플랫폼을 만들지 않는다.

```text
업무 설명 수집
→ 전사 확인
→ 업무 구조화
→ 적응형 딥인터뷰
→ 업무 지도 확인
→ 자동화·개선 후보 1~3개
→ G1 수준 개발 명세
```

실제 실행은 Zapier, Make, n8n, Power Automate, RPA, 커스텀 개발 등 외부 수단과 연결하는 후속 단계로 둔다.

## 12. 리서치 결론

Work Discovery AI의 위치는 다음 세 영역 사이에 있다.

```text
업무 개선 컨설팅
+
딥인터뷰 AI
+
자동화·프로그램 설계 에이전트
```

핵심 경쟁력은 자동화를 직접 실행하는 능력보다 다음에 있다.

> 사용자가 자동화 대상을 모르는 상태에서 실제 업무를 발견하고, 판단과 예외를 포함한 업무 모델로 변환한 뒤, 가장 적절한 개선·자동화 방식과 개발 명세를 생성하는 능력
