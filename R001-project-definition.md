# R001 — Project Definition

## 1. 프로젝트명

작업명: **Work Discovery AI**  
한국어 가칭: **업무발견 AI**

## 2. 프로젝트 배경

많은 개인과 기업은 AI 자동화가 업무에 도움이 될 수 있다는 사실은 알고 있지만, 실제 도입 단계에서는 무엇을 자동화해야 하는지, 자신의 업무를 자동화 관점에서 어떻게 설명해야 하는지, 실제 효과가 얼마나 될지를 판단하지 못해 시작하지 못한다.

기존 자동화 도구는 대체로 사용자가 자동화 대상을 이미 알고 있다는 전제에서 시작한다. 이 프로젝트는 그 이전 단계인 **업무 발견, 업무 구조화, 자동화 가능성 판단**부터 지원한다.

## 3. 해결하려는 핵심 문제

> 사용자가 자신의 업무를 자동화 관점에서 분석할 수 없기 때문에 AI 자동화에 접근하지 못하는 문제

시스템은 사용자가 자신의 일을 자연어와 음성으로 설명하도록 돕고, 반복 질문을 통해 업무를 구조화한 뒤 개선·자동화·업무 지원 프로그램으로 이어질 수 있는 후보를 제안한다.

## 4. 프로젝트 미션

> 어떠한 형태의 사용자든 자신이 하는 일을 설명하는 것만으로 AI 자동화 도입 가능성을 발견하고, 실제 업무 개선 또는 프로그램 제작까지 이어갈 수 있도록 돕는다.

## 5. 제품 비전

> 누구나 자신의 업무를 AI에게 설명하면, AI가 업무의 목적·흐름·판단·예외·도구·데이터를 이해하고, 자동화 또는 업무 지원이 가능한 부분을 찾아 실제 프로그램 설계와 개발 명세까지 생성하는 플랫폼

## 6. 핵심 사용자

### 6.1 1차 대상

- AI 자동화에 관심은 있지만 무엇을 자동화해야 할지 모르는 개인
- 반복적인 디지털 사무 업무를 수행하는 직원
- 별도 자동화 전문가가 없는 중소기업
- 업무 개선이 필요하지만 컨설팅 비용이 부담되는 팀
- 고객 업무를 분석해야 하는 자동화 개발자·컨설턴트

### 6.2 주요 사용자 특성

- 자신의 업무는 잘 알지만 시스템 설계 언어로 설명하지 못한다.
- 반복 업무를 당연한 것으로 받아들여 자동화 기회를 인식하지 못할 수 있다.
- 업무 중 판단과 예외를 명시적으로 정리한 경험이 적다.
- API, RPA, AI Agent 등의 차이를 모를 수 있다.
- 자동화 효과와 위험을 정량적으로 판단하기 어렵다.

## 7. 사용자 경험의 기본 구조

```text
1. 사용자가 자신의 역할과 업무를 설명한다.
2. 텍스트 입력 또는 음성 녹음을 사용한다.
3. 음성은 텍스트로 변환되고 사용자가 확인·수정한다.
4. AI가 설명에서 업무 구조를 추출한다.
5. AI가 빠진 정보와 모호한 내용을 반복 질문한다.
6. 사용자가 업무 지도를 확인하고 수정한다.
7. AI가 제거·단순화·표준화·자동화 후보를 분석한다.
8. 후보별 예상 효과, 구현 난이도, 위험, 사람의 역할을 제시한다.
9. 선택된 후보에 대해 실제 프로그램 또는 업무 지원 도구의 설계 명세를 만든다.
10. 결과를 개발자·Codex·자동화 도구에 전달할 수 있는 형태로 내보낸다.
```

## 8. 최종 산출물

최종 산출물은 단순한 인터뷰 요약이 아니라 다음 중 하나 이상이다.

- 불필요한 업무 제거안
- 업무 흐름 개선안
- 기존 도구 설정·활용안
- 규칙 기반 자동화
- RPA 자동화
- AI 보조 도구
- 승인형 AI Agent
- 업무 관리용 웹·앱 프로그램
- 기존 시스템 연동 프로그램
- 개발 가능한 PRD
- 화면 구조
- 데이터 모델
- API 설계
- 테스트 기준
- 개발 백로그
- Codex 등 개발 도구에 전달할 실행 명세

## 9. 성공 단계

### Level 1 — 업무 이해

AI가 사용자의 업무 목적, 흐름, 입력, 출력, 판단, 예외, 도구, 반복성을 구조화하고 사용자가 “내 업무를 정확히 이해했다”고 확인한다.

### Level 2 — 자동화 기회 분석

AI가 자동화 또는 개선 가능한 업무를 찾아 이유, 예상 효과, 구현 난이도, 위험을 설명한다.

### Level 3 — 프로그램 설계

AI가 선택된 후보에 대해 PRD, 기능 목록, 화면 흐름, 데이터 구조, API, 수용 기준을 생성한다.

### Level 4 — 개발 연결

AI가 개발자가 바로 구현하거나 Codex 등의 개발 에이전트에 전달할 수 있는 수준의 명세를 생성한다.

## 10. 핵심 설계 원칙

### 10.1 자동화보다 업무 이해가 먼저다

업무를 충분히 이해하지 않은 상태에서 자동화를 추천하지 않는다.

### 10.2 사용자의 설명을 그대로 사실로 확정하지 않는다

모든 정보는 사용자 발언, AI 추론, 다른 증거와의 일치, 사용자 최종 확인 상태를 구분한다.

### 10.3 자동화 전에 제거와 단순화를 검토한다

불필요하거나 중복된 업무는 자동화하지 않고 제거하거나 단순화한다.

### 10.4 업무 전체가 아니라 작업 단위로 분석한다

한 업무 안에서도 자동화 가능한 단계와 사람이 맡아야 할 단계를 분리한다.

### 10.5 판단과 예외를 핵심 데이터로 취급한다

단순 행동 기록뿐 아니라 판단 기준, 승인 조건, 예외 처리, 실패 복구 방법을 수집한다.

### 10.6 전면 자동화만을 목표로 하지 않는다

AI 초안, 추천, 승인형 실행, 대시보드, 검색, 알림 등 다양한 지원 수준을 고려한다.

### 10.7 근거와 추적성을 유지한다

모든 자동화 추천과 요구사항은 사용자의 원문 답변 또는 확인된 업무 모델과 연결되어야 한다.

## 11. 프로젝트 범위

### 포함

- 텍스트·음성 기반 업무 인터뷰
- 전사 확인
- 업무 구조화
- 적응형 딥인터뷰
- 업무 지도 검증
- 자동화·개선 후보 분석
- 프로그램 설계 명세 생성
- Markdown·JSON 등 구조화된 내보내기

### 초기 제외

- 외부 시스템에 직접 로그인하여 업무 실행
- 운영 코드 자동 배포
- 직원 상시 감시
- 회사 전체 프로세스 마이닝
- 고위험 의사결정의 완전 자율화
- ROI 보장
- 법적·규제 적합성 최종 판단

## 12. 프로젝트의 차별점

기존 자동화 제품은 일반적으로 다음 질문에서 시작한다.

> 무엇을 자동화하고 싶습니까?

이 프로젝트는 다음 질문에서 시작한다.

> 지금 어떤 일을 하고 있으며, 실제로 어떻게 처리하고 있습니까?

차별점은 자동화 제작 기능 자체보다 **자동화 기회를 발견하고 업무를 개발 가능한 수준으로 번역하는 능력**이다.

## 13. 핵심 시스템 구성

```text
Interview Engine
→ Work Modeling Engine
→ Automation Discovery Engine
→ Automation Reasoning Engine
→ Program Generation Engine
→ Evaluation & Validation
```

- **Interview Engine:** 빠진 정보와 모호성을 찾아 다음 질문을 생성한다.
- **Work Modeling Engine:** 답변을 업무, 작업, 행동, 판단, 규칙, 예외, 입력, 출력, 도구, 지표 구조로 변환한다.
- **Automation Discovery Engine:** 삭제, 단순화, 표준화, 자동화, AI 보조, Agent, 전용 프로그램 중 적합한 방향을 찾는다.
- **Automation Reasoning Engine:** 가치, 실행 가능성, 위험, 근거 신뢰도, 사람의 감독 수준을 평가한다.
- **Program Generation Engine:** PRD, 화면, 데이터, API, 테스트, 개발 백로그를 생성한다.
- **Evaluation & Validation:** 업무 이해도, 질문 품질, 추천 품질, 위험한 오추천, 명세 완성도를 평가한다.

## 14. MVP 방향

첫 MVP는 다음 흐름을 완성하는 데 집중한다.

```text
텍스트·음성 입력
→ 전사 확인
→ 업무 구조화
→ 적응형 질문
→ 사용자 업무 지도 확인
→ 자동화 후보 1~3개
→ 선택 후보의 G1 개발 명세
```

실제 프로그램 자동 생성과 운영 실행은 후속 단계로 분리한다.

## 15. 핵심 연구 질문

1. AI는 사용자의 업무를 어떤 구조로 저장해야 하는가?
2. AI는 어떤 질문을 어떤 순서로 해야 하는가?
3. AI는 언제 업무를 충분히 이해했다고 판단할 수 있는가?
4. 자동화, 단순화, AI 보조, 사람 유지 중 무엇이 적합한지 어떻게 판단할 것인가?
5. 사용자 답변에서 실제 개발 가능한 명세를 어떻게 생성할 것인가?
6. AI의 과도한 추론과 위험한 자동화 추천을 어떻게 방지할 것인가?
7. 사용자가 인터뷰 과정에서 이탈하지 않으면서도 충분한 정보를 얻으려면 어떻게 해야 하는가?
