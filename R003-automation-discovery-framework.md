# R003 — Automation Discovery Framework v1

## 1. 목적

ADF는 인터뷰로 수집된 업무 모델을 보고 다음 질문에 답한다.

1. 이 업무는 정말 필요한가?
2. 자동화보다 제거·통합·단순화·표준화가 먼저인가?
3. 어떤 해결 방식이 적합한가?
4. 어느 수준까지 AI 또는 시스템에 권한을 줘도 되는가?
5. 기대 효과는 어느 정도이며, 무엇이 불확실한가?

ADF의 결과는 “자동화 점수 82점” 같은 단일 숫자가 아니라, 비교 가능한 **판단 벡터와 근거 카드**다.

---

## 2. 판단 순서

### 2.1 가치 확인

- 업무 결과의 수신자 또는 고객은 누구인가?
- 이 단계가 만드는 가치 또는 통제 목적은 무엇인가?
- 이 단계를 없애면 실제로 어떤 문제가 생기는가?
- 관행 때문에 남아 있는 단계인가, 법·품질·보안상 필요한 단계인가?

가치가 확인되지 않으면 자동화 후보가 아니라 `ELIMINATE_OR_INVESTIGATE`로 분류한다.

### 2.2 현재 상태 개선

다음 순서로 검토한다.

1. `ELIMINATE` — 삭제
2. `COMBINE` — 중복 단계 통합
3. `REORDER` — 순서 변경
4. `SIMPLIFY` — 입력·승인·전환 축소
5. `STANDARDIZE` — 사람별 변형을 표준화
6. `CONFIGURE` — 기존 도구 설정 또는 교육
7. `INTEGRATE` — 시스템 간 데이터 연결
8. `AUTOMATE` — 규칙 기반 자동화/RPA
9. `ASSIST` — AI 보조
10. `AGENT` — 제한된 권한을 가진 에이전트
11. `CUSTOM_BUILD` — 전용 프로그램

자동화는 개선 수단 중 하나이며 항상 첫 번째 선택이 아니다.

### 2.3 해결 방식 선택

| 코드 | 해결 방식 | 대표 상황 |
|---|---|---|
| `NO_ACTION` | 유지 또는 추가 조사 | 가치·문제·빈도가 불명확 |
| `ELIMINATE` | 단계 삭제 | 결과가 사용되지 않음 |
| `SIMPLIFY_STANDARDIZE` | 절차 단순화·표준화 | 사람마다 방식이 다름 |
| `EXISTING_TOOL` | 기존 도구 설정·교육 | 이미 기능이 있으나 활용하지 않음 |
| `KNOWLEDGE_ASSIST` | 검색·가이드·초안 | 판단은 사람에게 남겨야 함 |
| `FORM_DASHBOARD` | 입력 폼·상태판 | 정보가 흩어지고 추적이 어려움 |
| `INTEGRATION_WORKFLOW` | API/웹훅 기반 흐름 | 시스템 간 반복 전송 |
| `DETERMINISTIC_AUTOMATION` | 규칙 엔진·RPA | 안정적이고 규칙이 명확 |
| `DOCUMENT_AI` | 추출·분류·검토 보조 | 비정형 문서가 주요 입력 |
| `AI_ASSISTANT` | 요약·추천·초안 | 예외와 판단이 많음 |
| `APPROVAL_AGENT` | 계획 후 승인받아 실행 | 다단계 디지털 작업, 권한 통제 가능 |
| `CUSTOM_APP` | 전용 웹/데스크톱 앱 | 데이터·상태·권한·업무 흐름을 함께 관리 |
| `HUMAN_ONLY` | 사람 중심 유지 | 책임·안전·맥락상 자율화 부적합 |

## 3. 다차원 평가 모델

### 3.1 Value Score, 0–100

- 처리량·빈도: 20
- 수작업 시간: 20
- 오류·재작업 비용: 15
- 지연·병목 영향: 10
- 고객/직원 불편: 10
- 규정·품질 통제 향상: 10
- 조직 확산 가능성: 10
- 전략적 중요성: 5

### 3.2 Feasibility Score, 0–100

- 입력의 디지털화: 10
- 입력의 구조화: 10
- 규칙 명확성: 15
- 예외 비율과 변형 수: 15
- 프로세스 안정성: 10
- 사용 앱·UI 안정성: 10
- API·내보내기 접근성: 10
- 테스트 가능한 완료 조건: 10
- 담당자·소유자 존재: 5
- 유지보수 가능성: 5

### 3.3 Risk Level, 0–4

최댓값 원칙을 사용한다. 평균으로 희석하지 않는다.

- 0: 무시 가능한 위험, 되돌리기 쉬움
- 1: 낮음, 내부 편의 업무
- 2: 중간, 승인·감사 로그 필요
- 3: 높음, 금전·법무·민감정보·대외 영향
- 4: 매우 높음, 생명·안전·중대한 권리·책임 결정

평가 영역:

- 개인정보·기밀정보
- 법적·규제 책임
- 재무 손실
- 안전·건강 영향
- 고객·평판 영향
- 권한 오남용
- 설명 가능성·감사 가능성
- 실패의 가역성

### 3.4 Evidence Confidence, 0.00–1.00

- 단순 사용자 주장만 존재: 최대 0.45
- 최근 실제 사례가 존재: 최대 0.65
- 사용자가 업무 지도를 확인: 최대 0.80
- 문서·화면·로그 등 독립 증거와 일치: 최대 0.95
- 운영 파일럿 데이터로 확인: 최대 1.00

모순이 해결되지 않았거나 입력 범위가 불명확하면 감점한다.

### 3.5 Oversight Level, 0–5

- 0: 정보 제공만
- 1: 초안 생성, 사람이 직접 사용
- 2: 추천, 사람이 결정
- 3: 승인 후 실행
- 4: 정책·금액·범위 제한 안에서 실행, 예외는 사람에게
- 5: 낮은 위험·가역적 작업에 한해 자동 실행

Risk 3 이상은 기본적으로 Oversight 3 이하로 제한한다. Risk 4는 MVP에서 자동 실행을 추천하지 않는다.

## 4. 하드 게이트

다음 중 하나라도 해결되지 않으면 “즉시 구축”을 추천하지 않는다.

- 업무 목표 또는 소유자가 없음
- 시작·종료 조건을 설명할 수 없음
- 개인정보·기밀정보·녹음 동의·보존 정책이 불명확
- 법적 승인이나 책임 주체가 불명확
- 현재 프로세스가 자주 바뀌며 표준 버전이 없음
- 핵심 입력이 디지털화되지 않았고 변환 방안도 없음
- 규칙과 예외가 문서화되지 않았으며 결과 오류가 중대함
- 처리량·시간·오류 비용을 추정할 근거가 전혀 없음
- 필요한 시스템 접근 권한을 얻을 수 없음
- 실패를 탐지하거나 되돌릴 방법이 없음

게이트 결과는 `BLOCKED`, `DISCOVERY_NEEDED`, `ENABLE_FIRST`, `READY_FOR_DESIGN` 중 하나다.

## 5. 효과 계산

### 5.1 기본 변수

- `annual_volume`
- `volume_scope`: 개인/팀/회사
- `average_handling_minutes`
- `error_rate`
- `average_rework_minutes`
- `automation_coverage`
- `expected_time_reduction`
- `loaded_hourly_cost`
- `verified_avoided_error_cost`
- `build_cost_range`
- `annual_run_cost_range`

### 5.2 계산식

```text
annual_manual_hours = annual_volume × average_handling_minutes ÷ 60
annual_rework_hours = annual_volume × error_rate × average_rework_minutes ÷ 60
recoverable_hours = (annual_manual_hours + annual_rework_hours)
                    × automation_coverage
                    × expected_time_reduction
annual_labor_benefit = recoverable_hours × loaded_hourly_cost
annual_total_benefit = annual_labor_benefit + verified_avoided_error_cost
net_year1_range = annual_total_benefit_range - build_cost_range - annual_run_cost_range
```

팀 전체 처리량에 이미 모든 직원의 건수가 포함되어 있다면 직원 수를 다시 곱하지 않는다. 모든 수치는 단일값보다 범위와 가정을 함께 표시한다.

## 6. 우선순위 포트폴리오

- `QUICK_WIN`: 가치 높음, 실행 가능성 높음, 위험 낮음
- `STRATEGIC_BET`: 가치 높음, 난이도 또는 변화관리 부담 높음
- `ENABLE_FIRST`: 데이터 정리·표준화·권한 확보가 선행되어야 함
- `HUMAN_CONTROLLED`: 효과는 있으나 사람 승인·감독이 필수
- `NO_BUILD`: 가치가 낮거나 기존 도구로 충분
- `DO_NOT_AUTOMATE`: 위험 또는 책임상 자율화 부적합

## 7. 자동화 후보 카드

각 후보는 다음 필드를 가져야 한다.

- 대상 업무·단계
- 현재 문제
- 추천 해결 방식과 대안
- 자동화/지원 범위
- 사람이 유지할 역할
- Value, Feasibility, Risk, Confidence, Oversight
- 효과 범위와 계산 가정
- 전제조건
- 실패 시 영향과 복구 방식
- 개인정보·권한·감사 요구
- 증거와 미해결 질문
- 최소 검증 실험
- MVP 기능
- 하지 않을 것

## 8. 초기 판단 규칙

1. 규칙이 명확하고 입력이 구조화·디지털이며 프로세스가 안정적이면 결정론적 워크플로 또는 RPA를 우선 검토한다.
2. 문서·메일처럼 비정형 입력이 있지만 출력 형식과 검토 기준이 명확하면 Document AI 또는 AI Assist를 검토한다.
3. 여러 시스템을 사용하지만 API가 있고 작업이 가역적이며 목표·권한 경계가 명확하면 승인형 에이전트를 검토한다.
4. 예외와 전문 판단이 많으면 전체 자동화보다 검색·요약·초안·추천을 우선한다.
5. UI만 접근 가능하고 UI가 안정적이면 RPA가 가능하지만, UI가 자주 바뀌면 API 또는 구조 개선을 먼저 찾는다.
6. 처리량과 효과가 낮으면 전용 개발보다 기존 도구 설정·템플릿·교육을 권한다.
7. 잘못된 프로세스는 자동화하지 않고 삭제·통합·단순화·표준화부터 한다.
8. 중대한 권리·안전·법적 책임 결정은 사람에게 최종 권한을 남긴다.

---

## 9. 미검증 가설

- 점수 가중치는 파일럿 전의 초기값이다.
- 직무별 가치와 위험 가중치가 달라질 수 있다.
- 사용자 인터뷰만으로 ROI를 확정할 수 없다.
- “자동화 가능”과 “실제 채택 가능”은 다르므로 소유자·변화관리·유지보수 지표가 필요하다.
