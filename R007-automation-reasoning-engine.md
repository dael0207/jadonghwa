# R007 — Automation Reasoning Engine v1

## 1. 목적

Automation Reasoning Engine은 확인된 업무 모델을 분석해 “무엇을 자동화할 수 있는가?”만 답하지 않는다. **어떤 문제를 어떤 방식으로 해결해야 하며, 사람과 시스템이 각각 무엇을 맡아야 하는가**를 설명한다.

결과는 하나의 정답이 아니라, 근거와 불확실성이 표시된 1~3개의 비교 가능한 후보이다.

---

## 2. 입력 조건

필수 입력:

- 확정된 업무 모델 스냅샷
- 핵심 필드 상태와 신뢰도
- 최근 실제 사례
- 판단·예외·권한 경계
- 처리량·시간·오류의 범위
- 데이터·도구·시스템 정보
- 사용자가 설정한 목표와 금지사항

조건 미충족 시 해결안을 억지로 생성하지 않고 `NEEDS_EVIDENCE` 또는 `ENABLE_FIRST`를 반환한다.

---

## 3. 분석 파이프라인

```text
1. 업무 모델 유효성 검사
2. 프로세스를 개선 가능한 단위로 분할
3. 가치·낭비·병목 분석
4. 삭제·통합·단순화·표준화 후보 생성
5. 기존 도구·연동·규칙 자동화·AI·에이전트·전용 앱 후보 생성
6. 하드 게이트 적용
7. 가치·실행 가능성·위험·증거 신뢰도 평가
8. 인간 감독 수준 할당
9. 효과 범위와 가정 계산
10. 1차 추천, 대안, 비추천 이유 생성
11. 최소 검증 실험 설계
12. 사용자 검토용 비교 카드 생성
```

---

## 4. 분석 단위

자동화는 프로세스 전체보다 아래 단위에서 판단한다.

- 한 단계의 행동
- 데이터 이동
- 반복 입력
- 문서 추출·분류
- 검색·조회
- 계산·비교
- 판단·승인
- 예외 처리
- 알림·추적
- 보고서 생성

각 단위는 `unit_id`, 입력, 출력, 규칙, 예외, 도구, 사람 역할, 처리량을 갖는다.

---

## 5. 해결안 생성 순서

### 5.1 제거·개선 후보

- 결과가 사용되지 않는 단계 삭제
- 같은 데이터를 여러 번 입력하는 단계 통합
- 승인 계층 축소
- 불필요한 형식 변환 제거
- 입력 양식 표준화
- 마스터 데이터 정리

### 5.2 기존 수단 후보

- 현재 도구의 미사용 기능
- 템플릿·매크로·필터·규칙
- 조직의 기존 자동화 플랫폼
- 교육·가이드·체크리스트

### 5.3 기술 후보

- API/웹훅 연동
- 규칙 기반 워크플로
- RPA
- 문서 AI
- 지식 검색·답변 보조
- AI 초안·추천
- 승인형 에이전트
- 전용 내부 도구
- 데이터 파이프라인·대시보드

## 6. 초기 매핑 규칙

| 업무 특성 | 우선 검토 방식 |
|---|---|
| 구조화된 디지털 입력, 규칙 명확, 예외 적음 | 결정론적 워크플로/API/RPA |
| 비정형 문서, 추출 필드·검토 기준 명확 | Document AI + 사람 검토 |
| 지식 검색·요약·초안이 중심 | AI Assistant |
| 여러 시스템, API 존재, 단계와 권한 경계 명확 | Approval Agent |
| 상태·권한·기록·협업을 함께 관리 | Custom App |
| 처리량 낮음, 기존 기능 존재 | Existing Tool/No Build |
| 프로세스 변형이 너무 많음 | Standardize/Case Management |
| 고위험 결정 또는 책임 판단 | Human-Controlled Assist |
| UI만 접근 가능하고 안정적 | RPA 검토 |
| UI가 불안정하고 API 없음 | Enable First 또는 비추천 |

## 7. 자율성 할당

### Level 0 — Observe

현황과 정보를 보여준다.

### Level 1 — Draft

초안·요약·분류 결과를 만들고 사람이 사용한다.

### Level 2 — Recommend

선택지와 근거를 제안하고 사람이 결정한다.

### Level 3 — Approve-to-Execute

실행 계획을 보여주고 사람이 승인하면 실행한다.

### Level 4 — Guardrailed Execution

금액·대상·횟수·정책 제한 안에서 실행하며 예외는 사람에게 보낸다.

### Level 5 — Autonomous Low-Risk

가역적이고 낮은 위험의 작업만 자동 실행한다.

자율성은 AI 능력보다 **실패 영향, 가역성, 권한, 감시 가능성**으로 결정한다.

## 8. 후보 평가

후보별로 다음을 출력한다.

```text
value_score: 0–100
feasibility_score: 0–100
risk_level: 0–4
evidence_confidence: 0.00–1.00
oversight_level: 0–5
portfolio_class: QUICK_WIN | STRATEGIC_BET | ENABLE_FIRST |
                 HUMAN_CONTROLLED | NO_BUILD | DO_NOT_AUTOMATE
```

점수는 후보 정렬을 돕는 보조 지표이며 하드 게이트를 무시하지 않는다.

---

## 9. 설명 의무

각 추천은 다음 질문에 답해야 한다.

- 왜 이 문제가 중요한가?
- 왜 이 해결 방식인가?
- 왜 더 단순한 방식은 부족한가?
- 왜 더 높은 자율성은 부적합한가?
- 사람은 무엇을 계속 해야 하는가?
- 어떤 증거가 추천을 지지하는가?
- 무엇을 아직 모르는가?
- 어떤 조건이 달라지면 추천이 바뀌는가?

---

## 10. ROI와 효과

효과는 낙관·기준·보수 시나리오로 제시한다.

- 절약 시간 범위
- 오류·재작업 감소 범위
- 처리 시간 감소
- 품질·추적성 개선
- 직원 경험 개선
- 구축·운영 비용 범위
- 가정과 데이터 출처

사용자가 수치를 제공하지 않은 경우 업계 평균을 사실처럼 넣지 않는다. 대신 “측정 필요”와 최소 측정 방법을 제안한다.

---

## 11. 최소 검증 실험

개발 전 각 후보에 최소 하나의 검증 실험을 붙인다.

예시:

- 20건의 샘플로 문서 추출 정확도 측정
- 2주간 처리 시간·오류율 기록
- 기존 도구의 규칙 기능으로 수동 프로토타입
- API 접근 가능 여부 확인
- 사용자 3명이 업무 지도를 검토
- 승인형 시뮬레이션으로 잘못된 실행률 측정
